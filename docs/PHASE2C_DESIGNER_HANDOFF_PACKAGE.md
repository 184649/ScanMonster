# Phase 2C Designer Handoff Package

Status: External handoff assembly guide / design only / 2026-07-18

## 1. Purpose

This guide assembles the smallest package that lets an external human designer understand WORLDAWN's product purpose, desired character quality, constraints, stages, deliverables, and acceptance process without being anchored to failed AI exploration. It is not a production-data export and does not assign official names or character IDs.

## 2. Recommended explanation order

1. Explain the barcode/QR discovery encounter in one minute.
2. Explain the desired feeling: discover, remember, record, and meet again—not combat power.
3. State the commercial goal and prohibition on concrete imitation.
4. Review the rarity philosophy, emphasizing that normal is not lower-quality art.
5. Review the project-wide Phase 2C brief.
6. Review the three flagship categories and motif-replacement conditions.
7. Review Stage 0 and Stage 1 as the first paid contract unit.
8. Review later deliverables and acceptance gates only to show the intended path.
9. Confirm feedback ownership, schedule, revisions, tools, rights, confidentiality, and AI disclosure.
10. Provide only the minimal product/UI context needed to understand use sizes and discovery moments.

## 3. Files and information to provide

### Core external documents

- `PHASE2C_HUMAN_CHARACTER_DESIGNER_BRIEF.md` — standalone project and IP brief.
- `PHASE2C_FLAGSHIP_CHARACTER_BRIEFS.md` — three structural tests and failure modes.
- `PHASE2C_DELIVERABLES_AND_ACCEPTANCE.md` — Stage 0–6 outputs, approvals, and acceptance.
- The executed statement of work and applicable portions of `PHASE2C_CONTRACT_AND_RIGHTS_CHECKLIST.md`.

### Product context

- WORLDAWN overview and central discovery vocabulary.
- Barcode/QR scan → encounter → official discovery number → discovery certificate → dex/history/calendar → rediscovery/title flow.
- Commercial goal: a memorable independent collection IP whose characters users want to discover.
- Rarity order and role: `normal`, `rare`, `legendary`, `secret`; rarity is not a design-quality ladder.
- Existing IDs, official discovery numbers, certificates, and histories are preserved and are not design variables.
- The future quality-first portfolio recommendation is 24 characters, but no expansion begins before the flagship baseline passes.

### Initial organism candidates

These are motif candidates, not approved characters or production IDs:

| Structural category | Default | Other candidates for discussion |
|---|---|---|
| Small, approachable ground animal | Hamster | Hedgehog, prairie dog |
| Large ground animal with presence | Elephant | Rhinoceros, bison |
| Non-mammal / different body | Snake | Chameleon, gecko |

Any change must satisfy the formal replacement conditions and receive user approval. Sheep is excluded.

### Minimal app/UI context

Provide a short flow description or only the smallest set of screens needed to show:

- the scan action and encounter reveal;
- where the official discovery number and certificate appear;
- the dex/list thumbnail context at 64 px and 128 px;
- the individual record/history view;
- the calendar or rediscovery context.

UI is context, not a style reference. Remove personal data, unpublished secrets, raw scan codes, product data, OCR text, precise location, credentials, analytics identifiers, and unrelated internal screens before sharing. This package does not change UI.

## 4. Material not provided at the start

As a rule, do not initially provide:

- large collections of failed AI images;
- old, long generation prompts;
- detailed histories of prompt corrections and failed cleanup;
- images that anchor the designer to an existing rejected option;
- boards of famous-IP images offered as imitation references;
- production workbooks, master JSON, classification data, catalogs, server seeds, database dumps, or user records;
- repository credentials, internal secrets, or source code unrelated to the visual use case.

The designer should respond to the product, character questions, and acceptance requirements—not average prior failed outputs.

## 5. Conditional use of AI exploration records

AI exploration images may be shared only if the designer specifically asks to understand failure patterns and the rights/confidentiality review permits it. Supply the minimum examples with this written legend:

> These are failure records, not correct examples, approved candidates, style references, or imitation targets. Do not reproduce them. They show directions that converged on generic animals, pose/orientation/body variations, pictograms, or one repeated face. Use them only to avoid returning to those directions.

At the start of Phase 2C, no Phase 2B hamster, elephant, or snake board exists in the repository. Do not imply that missing files are approved, available, or hidden candidates.

## 6. Handoff checklist

Before sending:

- [ ] The recipient and confidentiality terms are confirmed.
- [ ] The brief can be read without internal repository context.
- [ ] The project is clearly discovery-centered and not presented as combat-centered.
- [ ] Default motifs and replacement approval are clear.
- [ ] Stage 0/1 are identified as the first paid unit.
- [ ] The exact deliverable count, price, timing, revisions, and acceptance are in the signed scope.
- [ ] 64 px, 128 px, silhouette, toy, and existing-IP checks are included.
- [ ] The user—not automation—owns preference decisions.
- [ ] Rights, source files, third-party materials, AI use/training, and portfolio disclosure are addressed.
- [ ] Shared UI material is minimal and sanitized.
- [ ] No failed image is labeled as a correct or style reference.
- [ ] No production ID, name, rarity, release status, or image path is assigned.

## 7. Receipt and version record

Record package version, send date, recipient, file list, file hashes or stable links, confidentiality status, contract/SOW version, known omissions, questions, and subsequent replacements. If a file changes, send a clearly versioned replacement and preserve the approval trail.
